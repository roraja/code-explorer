# 10 Ideas to Improve Code Explorer

> **Date:** 2026-03-29
> **Context:** After a thorough review of the full codebase — extension host, webview, analysis pipeline, cache layer, prompt strategies, LLM providers, UI rendering, and design docs — these are 10 concrete improvement ideas to help developers better understand code.

---

## 1. Cross-Symbol Dependency Graph View

**Problem:** Today, each symbol is analyzed in isolation. A developer exploring a complex feature must mentally stitch together dozens of individual symbol tabs to understand how components relate. There is no way to visualize the "big picture" — how a request flows through `extension.ts` → `CodeExplorerViewProvider` → `AnalysisOrchestrator` → `CacheStore` → `LLMProvider`.

**Idea:** Add a "Dependency Graph" view mode that renders an interactive, navigable graph of all analyzed symbols in the workspace. When a user explores a symbol, it becomes a node in the graph, and its sub-functions, callers, dependencies, and related symbols become connected nodes. Clicking any node opens its analysis tab.

**How it works:**
- Leverage the existing `relatedSymbols`, `subFunctions`, `callStacks`, and `dependencies` fields already present in `AnalysisResult`
- Build an in-memory graph structure from all cached analyses (the cache already stores these relationships)
- Render using a force-directed layout library (e.g., D3-force or ELK.js) in the webview, or as a large Mermaid graph diagram
- Support filtering by: file, symbol kind, call depth, or specific dependency chains
- Allow "trace mode" — select two symbols and highlight all paths between them

**Why it matters:** The #1 pain point in the PRD is "Understanding 'what calls what' is manual and slow." A visual graph directly addresses this. The data is already being generated by the LLM — it just needs to be connected across analyses.

**Modules affected:** `webview/src/main.ts` (new graph renderer), `CodeExplorerViewProvider.ts` (new message type), `CacheStore.ts` (graph query method to walk all cached analyses), new `src/graph/GraphBuilder.ts`.

## 2. "Explain This Change" — Diff-Aware Analysis

**Problem:** Code Explorer currently analyzes symbols as they exist at a point in time. But developers often need to understand *what changed* — during code review, after a pull, or when debugging a regression. The question isn't "what does this function do?" but "what did this change *do* to this function, and what might it break?"

**Idea:** Add a "diff-aware analysis" mode that compares the current version of a symbol against its cached analysis (or a git baseline), identifies what changed, and generates an impact assessment.

**How it works:**
- When a symbol has a cached analysis and the source has changed (detected via `sourceHash` in `AnalysisMetadata`, which is already defined but always empty `''`), offer a "What Changed?" action
- Implement `HashService` (already planned in docs but not built) to compute SHA-256 of symbol source
- Build a new `PromptBuilder.buildDiffAnalysis(oldAnalysis, newSource)` prompt that feeds the LLM both the previous analysis and the new source code
- The LLM response includes: what changed (added/removed/modified logic), impact on callers, new potential issues, whether the existing cache entries for callers/dependencies are now stale
- Display as a dedicated "Changes" section in the analysis tab, with before/after summaries

**Why it matters:** This bridges the gap between static "what is this?" analysis and the dynamic "what happened?" question that dominates real development work (code review, post-merge understanding, debugging regressions).

**Modules affected:** New `HashService` in `src/cache/`, `PromptBuilder.ts` (new `buildDiffAnalysis` method), `AnalysisOrchestrator.ts` (diff-aware path), `ResponseParser.ts` (new diff response format), `webview/src/main.ts` (diff display section).

## 3. Wire Static Analysis Into the Pipeline (Ground-Truthing)

**Problem:** `StaticAnalyzer.ts` has three powerful methods — `findReferences()`, `buildCallHierarchy()`, `getTypeHierarchy()` — that are **fully implemented but never called** by the analysis pipeline. The orchestrator relies entirely on LLM-generated data for callers, references, and relationships. LLMs hallucinate: they invent callers that don't exist, miss references that do, and guess at type hierarchies. The static data from VS Code's language server is ground truth, but it's sitting unused.

**Idea:** Merge static analysis results with LLM results to produce analysis that is both semantically rich (LLM) and factually accurate (static).

**How it works:**
- In `AnalysisOrchestrator.analyzeSymbol()`, call `findReferences()` and `buildCallHierarchy()` in parallel with the LLM call (they're already implemented, just not wired)
- In `analyzeFromCursor()`, after the LLM resolves the symbol identity, run static analysis on the resolved `SymbolInfo`
- Merge strategy: use static data as the authoritative source for `callStacks`, `usages`, and `relationships`, then layer LLM-generated semantic descriptions on top (the LLM's "context" field for callers, the "description" for relationships)
- Add a `"source": "static" | "llm" | "merged"` field to `CallStackEntry` and `UsageEntry` so the UI can badge which data is verified vs. AI-generated
- In the webview, show a small icon (e.g., checkmark for static, sparkle for LLM) next to each caller/reference entry

**Why it matters:** This is the lowest-hanging fruit for accuracy improvement. The code is written, tested, and ready — it just needs to be plugged in. It eliminates the entire class of "LLM hallucinated a caller that doesn't exist" bugs.

**Modules affected:** `AnalysisOrchestrator.ts` (wire calls), `types.ts` (add `source` field), `webview/src/main.ts` (source badges), `ResponseParser.ts` (merge logic).

## 4. Hover Cards — Inline Analysis Preview Without Leaving the Editor

**Problem:** The current workflow is disruptive: place cursor on symbol → press Ctrl+Shift+E → switch focus to sidebar → read analysis → switch back to editor. For quick questions ("what does this function return?" or "who calls this?"), this context switch is too heavy. The `CodeExplorerHoverProvider` is planned in the design docs but not implemented.

**Idea:** Add a hover card that appears when the user hovers over a symbol in the editor. If the symbol is already cached, show an instant inline preview (overview, return type, caller count). If not cached, show a "Click to analyze" prompt. Clicking the hover card opens the full analysis in the sidebar.

**How it works:**
- Register a `vscode.HoverProvider` that intercepts hover events on supported languages
- On hover, check `CacheStore.findByCursor()` for a cached analysis (this is already fast — no LLM call needed)
- If cached: render a compact Markdown hover card with: overview (first sentence), symbol kind badge, return type, caller count, "Open in Code Explorer" link
- If not cached: render "No analysis yet — Ctrl+Shift+E to explore"
- Add a config `codeExplorer.showHoverCards` (already defined in `constants.ts` as `CONFIG.SHOW_HOVER_CARDS`) to let users toggle this
- Hover cards are read-only and lightweight — no LLM calls triggered on hover

**Why it matters:** This reduces the friction of using Code Explorer from "explicit command invocation" to "glance at it while reading code." For cached symbols (which grow over time), this turns Code Explorer into an always-available knowledge layer that's visible without any extra action.

**Modules affected:** New `src/providers/CodeExplorerHoverProvider.ts`, `extension.ts` (register provider), `CacheStore.ts` (already has `findByCursor`).

## 5. Exploration History & Breadcrumb Trail

**Problem:** When exploring a complex call chain, developers click through symbol → sub-function → sub-sub-function → type definition, opening many tabs. After going 5 levels deep, they lose track of *how* they got there and *why* they started exploring. The tab bar shows open symbols but not the navigation path. There's no "back" button, no trail, no way to retrace the investigation.

**Idea:** Add an exploration history that tracks the navigation path as a breadcrumb trail, with back/forward navigation and the ability to "bookmark" important symbols in a named investigation.

**How it works:**
- Track a navigation stack in `CodeExplorerViewProvider`: every time `openTab()` or `openTabFromCursor()` is called from a user action, push `{ fromTabId, toTabId, trigger }` onto a history stack
- Render a breadcrumb bar above the analysis content: `AnalysisOrchestrator → analyzeSymbol() → CacheStore.read() → ...`
- Add back/forward buttons (like browser navigation) that re-activate previous tabs without re-analyzing
- Add "Pin Investigation" — user can name a trail (e.g., "Tracing the cache miss bug") and save it as a persistent workspace bookmark
- Integrate with the existing `TabSessionStore` for persistence across reload

**Why it matters:** Code exploration is inherently a *journey*, not a collection of isolated lookups. The current tab model loses the narrative of the investigation. Breadcrumbs + history turn Code Explorer from a reference tool into an exploration companion.

**Modules affected:** `CodeExplorerViewProvider.ts` (history stack, breadcrumb state), `webview/src/main.ts` (breadcrumb bar UI), `TabSessionStore.ts` (persist investigations), `types.ts` (new `NavigationEntry` type).

## 6. Smart Auto-Explore on File Open / Scroll

**Problem:** Code Explorer is purely on-demand: nothing happens until the user explicitly triggers Ctrl+Shift+E. This means the cache is always empty for new files, and the first exploration is always slow (waiting for the LLM). Developers working in a file for 30 minutes have no analysis cached for any of its symbols — until they manually request each one.

**Idea:** Automatically pre-analyze symbols as the developer works. When a file is opened or scrolled, silently queue analysis for visible/important symbols in the background. By the time the developer wants to explore a symbol, it's already cached.

**How it works:**
- Use `vscode.window.onDidChangeVisibleTextEditors` and `vscode.window.onDidChangeTextEditorVisibleRanges` to detect which symbols are on screen
- Debounce (5-10 seconds of stable visible range) before queueing
- Use the existing `analyzeFile()` method in `AnalysisOrchestrator` — it already analyzes all symbols in a file in a single LLM call
- Queue at `PRIORITY_BACKGROUND` (already defined in constants as `1`, vs `10` for user-triggered) so explicit user requests always preempt
- Implement the `AnalysisQueue` with priority (planned but not built) to prevent background work from blocking foreground work
- Add rate limiting: max 1 background analysis per 60 seconds, only for files the user has had open for >10 seconds
- Add config `codeExplorer.autoAnalyzeOnSave` (already defined in constants) and `codeExplorer.autoAnalyzeOnView` (new)

**Why it matters:** The best analysis is the one that's ready before you ask for it. Pre-caching transforms Code Explorer from "slow-on-first-use" to "instant-always." The `analyzeFile()` method already exists — this just triggers it intelligently.

**Modules affected:** New `src/analysis/BackgroundAnalyzer.ts`, `extension.ts` (register event listeners), `AnalysisOrchestrator.ts` (queue integration), new `src/analysis/AnalysisQueue.ts`.

## 7. Multi-Language Support with Language-Aware Prompts

**Problem:** `SUPPORTED_LANGUAGES` in `constants.ts` only lists `typescript`, `javascript`, `typescriptreact`, `javascriptreact`. The `_guessLanguage()` method in `PromptBuilder.ts` can detect Python, Java, C#, C/C++, but the extension doesn't officially support them — and the prompt strategies are written with TypeScript/JavaScript idioms (references to `const`, `let`, `interface`, `Promise<T>`). A Python developer exploring a `class` would get prompts asking about TypeScript-specific patterns.

**Idea:** Expand language support to Python, Go, Rust, Java, C#, and C/C++ with language-aware prompt strategies that use idioms natural to each language.

**How it works:**
- Expand `SUPPORTED_LANGUAGES` to include `python`, `go`, `rust`, `java`, `csharp`, `cpp`, `c`
- Create language-specific prompt modifiers that adjust terminology: "methods" vs "functions" vs "associated functions" (Rust), "interface" vs "trait" vs "protocol", "property" vs "field" vs "attribute"
- Adjust `FunctionPromptStrategy` to mention language-specific patterns: Python decorators, Go goroutines, Rust ownership/borrowing, Java annotations
- The LLM already handles multi-language — it's the prompts that need localization, not the LLM provider
- Adjust `WATCHED_EXTENSIONS` to include `.py`, `.go`, `.rs`, `.java`, `.cs`, `.cpp`, `.c`, `.h`
- Test with the sample-workspace: add a Python and Go sample alongside the existing C++ sample

**Why it matters:** Code Explorer's value proposition ("understand any codebase") is limited to JS/TS today. Most enterprise teams work across multiple languages. The LLM can already analyze any language — the extension just needs to stop blocking non-JS/TS files and tune its prompts.

**Modules affected:** `constants.ts` (expand supported languages), `PromptBuilder.ts` (language-aware modifiers), `src/llm/prompts/*.ts` (language hints in strategies), `package.json` (activation events for new languages).

## 8. "Compare Two Symbols" Side-by-Side

**Problem:** Developers frequently need to understand the *differences* between two similar symbols: two implementations of the same interface, two versions of a refactored function, or two competing approaches in a codebase. Today, they can open two tabs in the sidebar, but must visually compare them by switching back and forth. There's no structured comparison.

**Idea:** Add a "Compare" action that takes two analyzed symbols and generates a structured side-by-side diff of their behavior, showing similarities and differences in purpose, logic, parameters, callers, and potential issues.

**How it works:**
- User selects two tabs (or right-clicks a tab and chooses "Compare with...") to trigger comparison
- Build a new `PromptBuilder.buildComparison(symbolA, analysisA, symbolB, analysisB)` that sends both analyses + both source codes to the LLM
- The LLM produces: shared purpose, behavioral differences, parameter differences, performance tradeoffs, which callers use which, recommendation on consolidation
- Render as a new tab type (`status: 'comparison'`) with a two-column layout in the webview
- Especially useful for: understanding polymorphic implementations, comparing before/after refactors, evaluating competing utility functions

**Why it matters:** "How is X different from Y?" is one of the most common questions when reading code, especially in large codebases with similar-looking components. No existing tool answers this question structurally.

**Modules affected:** `PromptBuilder.ts` (new `buildComparison` method), `ResponseParser.ts` (comparison response format), `types.ts` (new `ComparisonResult` type, new tab status), `CodeExplorerViewProvider.ts` (comparison trigger), `webview/src/main.ts` (two-column comparison renderer).

## 9. Stale Cache Detection via File Watcher

**Problem:** The `AnalysisMetadata` has fields for `sourceHash` and `stale`, and the `WATCHED_EXTENSIONS` glob is defined — but no file watcher exists. When a developer modifies a function, its cached analysis becomes silently wrong. The developer has no way to know that the overview, sub-functions, and callers in the sidebar are outdated. Worse, the pre-cached related symbols become wrong too, creating a cascade of stale data.

**Idea:** Implement the file watcher → cache invalidation pipeline. When a source file is saved, mark all cache entries for symbols in that file as stale. In the webview, show stale entries with a visual indicator and a one-click "Re-analyze" button.

**How it works:**
- Register a `vscode.workspace.createFileSystemWatcher(WATCHED_EXTENSIONS)` in `extension.ts`
- On file change: scan the cache directory for files under that source path (e.g., all `.md` files under `.vscode/code-explorer/src/analysis/AnalysisOrchestrator.ts/`)
- Set `metadata.stale = true` in each affected cache file's YAML frontmatter (the field already exists and is already checked by the orchestrator)
- In the webview, when `metadata.stale` is true, show a yellow "Stale — source has changed" banner with a "Re-analyze" button
- Also mark dependent analyses as stale: if `FunctionA` calls `FunctionB` and `FunctionB`'s file changes, `FunctionA`'s callers list may be wrong. Use `dependentFileHashes` in `AnalysisMetadata` (already defined, always empty) to track cross-file dependencies
- Implement `HashService` to compute and store SHA-256 hashes during `CacheStore.write()`

**Why it matters:** Stale analysis is worse than no analysis — it gives developers false confidence. This is the #1 data integrity issue in the current implementation. The metadata fields exist, the orchestrator already checks `stale`, and the webview already has a `'stale'` tab status. The only missing piece is the trigger.

**Modules affected:** `extension.ts` (register file watcher), `CacheStore.ts` (mark stale, hash comparison), new `src/cache/HashService.ts`, `webview/src/main.ts` (stale banner already partially handled).

## 10. Contextual "Why?" Annotations on Code

**Problem:** Code Explorer shows analysis in a sidebar — physically separated from the code. The developer must constantly cross-reference: read a step in the sidebar ("Step 3: validates the input against the schema"), then scan the actual code to find which lines correspond. For complex functions with 50+ lines, this mental mapping is exhausting.

**Idea:** Add inline CodeLens annotations directly in the editor that show contextual "why" labels above key lines, sourced from the cached analysis. When a symbol has been analyzed, its function steps, data flow points, and potential issues appear as subtle inline labels right where they happen in the code.

**How it works:**
- Implement `CodeExplorerCodeLensProvider` (planned in design docs, not built)
- When a file has cached analyses, provide CodeLens items at the start of each analyzed function/class
- Show the overview as a one-line CodeLens above the symbol definition: `// Code Explorer: Coordinates cache check + LLM analysis pipeline...`
- For functions with `functionSteps`, add CodeLens at the approximate line of each step: `// Step 2: Reads symbol source code for the LLM prompt`
- For variables with `dataFlow`, add CodeLens at each flow point: `// Data: created here, consumed at line 45`
- For symbols with `potentialIssues`, show a warning CodeLens: `// Issue: No error handling for cache write failure`
- All CodeLens items are clickable — clicking opens the full analysis in the sidebar
- Add a config `codeExplorer.showCodeLens` (boolean, default false) since some users will find inline annotations distracting

**Why it matters:** This brings the analysis *to the code* instead of asking the developer to go to the analysis. It's the difference between a reference book (sidebar) and annotated source (inline). For the "understanding code" use case, contextual annotations at the point of relevance are dramatically more effective than a separate panel.

**Modules affected:** New `src/providers/CodeExplorerCodeLensProvider.ts`, `extension.ts` (register provider), `CacheStore.ts` (query by file path), `constants.ts` (new config key).
